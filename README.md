# Provenance Guard

A backend system any creative-sharing platform can plug into to **classify whether submitted text was written by a human or generated by AI**, score confidence in that classification, surface a plain-language transparency label to readers, and let creators appeal a decision they disagree with.

The goal is not to police creativity. Perfect AI detection is an unsolved problem, so the system is built to **communicate uncertainty honestly** and to **protect creators from false accusations** rather than to deliver a confident binary verdict.

## Quick start

```bash
python -m venv .venv
source .venv/Scripts/activate      # Windows (Git Bash)
# .venv\Scripts\activate           # Windows (Command Prompt)
# source .venv/bin/activate        # Mac/Linux

pip install -r requirements.txt
```

Create a `.env` file in the repo root (it is git-ignored — never commit it):

```
GROQ_API_KEY=your_key_here
```

Run the server:

```bash
python app.py
# Serves on http://localhost:5000
```

> **No API key?** The system still runs. If `GROQ_API_KEY` is missing, the LLM signal returns a neutral `0.50` and the two pure-Python signals still produce a result, so the pipeline degrades gracefully instead of crashing.

## Architecture overview

The path a single submission takes from input to the label a reader sees:

```
Client / platform
   │  POST /submit { text, creator_id }
   ▼
Flask validation  ──(rejects empty fields / text < 40 chars)
   │  raw text
   ▼
Detection pipeline (3 signals run on the same text)
   ├── Signal 1: Groq LLM judge          → ai_score 0–1 + reason
   ├── Signal 2: Stylometric heuristics   → ai_score 0–1 + metrics
   └── Signal 3: Phrase-pattern heuristic → ai_score 0–1 + metrics
   │  three independent ai_scores
   ▼
Confidence scoring (weighted ensemble → combined ai_score + confidence)
   │  combined score
   ▼
Transparency label generator  → exact reader-facing text
   │
   ▼
Submission store (data/submissions.json) + Audit logger (logs/audit.jsonl)
   │
   ▼
JSON response to client
```

**Appeal flow:** `POST /appeal` looks up the original decision in the submission store, flips its status from `classified` to `under_review`, writes an appeal entry into the audit log *beside* the original classification, and returns a confirmation. (Automated re-classification is intentionally not performed — a human reviewer makes the final call.)

A fuller narrative and ASCII diagram live in [planning.md](planning.md#architecture).

## Detection signals

The pipeline uses **three distinct signals** (two are required; the third is the [ensemble stretch](#stretch-feature-ensemble-detection)). They are independent because they measure different *kinds* of property — one semantic, one structural, one lexical.

### Signal 1 — Groq LLM judge (semantic)
- **What it measures:** A holistic read of whether the text *feels* AI-generated — generic phrasing, template-like coherence, suspicious smoothness — returned as `ai_score` 0–1 plus a one-line reason. Uses `llama-3.3-70b-versatile` at `temperature=0`.
- **Why I chose it:** It captures semantic and stylistic cues that are nearly impossible to encode as rules.
- **What it misses:** It can over-trust polished human writing and under-detect lightly edited AI output. It is a judgment, never proof of authorship. In testing it also tends to compress its range into ~0.2–0.85.

### Signal 2 — Stylometric heuristics (structural)
- **What it measures:** Measurable text statistics — sentence-length variance, average sentence length, type-token ratio (vocabulary diversity), and punctuation density. AI text tends to be more *uniform*; human writing is more irregular.
- **Why I chose it:** It is fully independent of the LLM because it uses arithmetic on the text, not semantic judgment, so it can disagree with the LLM in informative ways.
- **What it misses:** Formal human writing (academic, legal) looks uniform and scores AI-like; short text gives weak statistics because both human and AI samples have high vocabulary diversity when short.

### Signal 3 — Phrase-pattern heuristic (lexical)
- **What it measures:** Counts of stock AI connective phrases ("it is important to note," "paradigm shift," "delve into," "rich tapestry"), offset by casual human markers ("lol," "tbh," "ok so").
- **Why I chose it:** It cheaply catches a specific failure mode — generic AI essay register — that stylometrics alone can miss.
- **What it misses:** Trivially fooled by avoiding stock phrases, and it can over-flag genuine human academic writing that legitimately uses formal transitions.

## Confidence scoring

Each signal emits an `ai_score` in `[0, 1]` (`0` = strongly human, `1` = strongly AI). They are combined with a fixed **weighted ensemble**:

```
combined_ai_score = 0.50 * llm_judge
                  + 0.30 * stylometric
                  + 0.20 * phrase_pattern
```

The LLM gets the most weight (holistic context), stylometrics second (independent and measurable), phrase-pattern least (useful but brittle and easy to game).

Confidence is then how far the combined score sits from the "no idea" midpoint:

```
confidence = max(combined_ai_score, 1 - combined_ai_score)
```

So a score near `0.5` yields low confidence (~0.5), while scores near `0` or `1` yield high confidence. The combined score maps to a label via **asymmetric thresholds**:

| Combined `ai_score` | Attribution | Label |
|---:|---|---|
| `0.00 – 0.40` | `likely_human` | High-confidence human |
| `0.41 – 0.69` | `uncertain` | Uncertain |
| `0.70 – 1.00` | `likely_ai` | High-confidence AI |

### How I validated that scores are meaningful
I ran the four canonical test inputs (clearly AI, clearly human, formal-human borderline, edited-AI borderline) plus an extreme AI sample, printing each signal's score separately so I could see *which* signal drove each result. The scores ordered exactly as intuition predicts, and the signals visibly disagree on borderline cases (which is the point of an ensemble):

| Input | llm | stylo | phrase | **combined** | label |
|---|---:|---:|---:|---:|---|
| Casual human ("ramen place…") | 0.20 | 0.30 | 0.00 | **0.19** | likely_human |
| Edited-AI borderline | 0.35 | 0.40 | 0.00 | **0.30** | likely_human |
| Formal-human borderline | 0.60 | 0.53 | 0.00 | **0.46** | uncertain |
| Generic AI essay | 0.82 | 0.39 | 0.75 | **0.68** | uncertain |
| Extreme AI "slop" | 0.82 | 0.50 | 0.78 | **0.72** | likely_ai |

This testing surfaced a real calibration problem: the stylometric signal rarely exceeds ~0.55 on short text, so a `0.75` AI threshold was **unreachable** — even blatant AI slop capped at `0.72`. I lowered the `likely_ai` threshold to `0.70` so the label is achievable, and documented the change in [planning.md](planning.md). The asymmetry (generous human band, strict AI band) is intentional: see [Known limitations](#known-limitations).

### Two example submissions with noticeably different confidence

**High-confidence case** — casual human writing:
```json
{ "creator_id": "alice", "attribution": "likely_human",
  "ai_score": 0.1897, "confidence": 0.8103,
  "signals": { "llm_judge": 0.20, "stylometric_heuristics": 0.2991, "phrase_pattern_heuristic": 0.0 } }
```

**Lower-confidence case** — formal human writing about monetary policy:
```json
{ "creator_id": "carol", "attribution": "uncertain",
  "ai_score": 0.4582, "confidence": 0.5418,
  "signals": { "llm_judge": 0.60, "stylometric_heuristics": 0.5274, "phrase_pattern_heuristic": 0.0 } }
```

Confidence `0.81` vs `0.54` produces two different labels and two different reader experiences — exactly the meaningful variation the score is meant to express.

## Transparency label

The label is written for a **non-technical reader** — no "classifier," no "score," no jargon — and it never claims to *prove* authorship. The exact text of all three variants:

| Variant | Exact label text |
|---|---|
| **High-confidence AI** | "This submission appears likely AI-generated. Our system found strong AI-like patterns, but this is not a final judgment; the creator may appeal." |
| **High-confidence human** | "This submission appears likely human-written. Our system found mostly human-like patterns, but no automated check can prove authorship." |
| **Uncertain** | "We are not confident enough to label this submission as AI- or human-written. It will be shown without a strong attribution claim, and the creator can provide more context." |

The label text changes with the result — it is not the same string with a different number attached. All three are reachable with real inputs (`alice` → human, `carol` → uncertain, `bob` → AI in the testing above).

## Appeals workflow

A creator who believes they were misclassified sends `POST /appeal` with the `content_id` and their `creator_reasoning`. The system:

1. Looks up the original content record.
2. Updates its status `classified → under_review`.
3. Stores the reasoning on the record.
4. Writes an appeal entry to the audit log **beside** the original classification (original attribution, score, and confidence are preserved).
5. Returns a confirmation.

A human reviewer opening the queue sees the original text preview, attribution, confidence, all three individual signal scores, and the creator's reasoning — enough context to make the final call. Re-classification is deliberately left to the human.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "672c33ec-...", "creator_reasoning": "I am an economics PhD student and wrote this from my own research..."}'
# → { "status": "under_review", "message": "Your appeal was received and is now under review.", ... }
```

## Rate limiting

Applied to `POST /submit` via Flask-Limiter (in-memory storage, keyed per client IP):

```python
@limiter.limit("10 per minute;100 per day")
```

**Reasoning, tied to realistic writing-platform usage:**
- **10 / minute** — A genuine creator submits a poem or essay occasionally, not ten times a minute. This cap is invisible to real use but immediately stops a script trying to flood the detector or brute-force inputs to reverse-engineer the thresholds.
- **100 / day** — Generous for any human (even a prolific writer revising drafts) while bounding the worst-case cost of a single client hammering the paid Groq API over a day.

**Evidence** — 12 rapid requests against the 10/minute limit (3 submissions had already run that minute, so the 8th request here is the 11th overall):

```
request 1 -> 200      request 7  -> 200
request 2 -> 200      request 8  -> 429
request 3 -> 200      request 9  -> 429
request 4 -> 200      request 10 -> 429
request 5 -> 200      request 11 -> 429
request 6 -> 200      request 12 -> 429
```

## Audit log

Every decision is appended as one structured JSON object per line to `logs/audit.jsonl` (JSONL was chosen because it is append-only, structured, and processable line-by-line by production log tooling). Submission entries record timestamp, content/creator IDs, attribution, combined score, confidence, every individual signal score, the label, and a text preview. Appeal entries record the original decision alongside the creator's reasoning and the new status.

Read it back any time with `GET /log`. A real sample below — a classification, and the appeal filed against it (signal scores abbreviated to `ai_score` per signal for readability):

```json
{"timestamp": "2026-06-30T23:07:45.514694+00:00", "event_type": "submission_classified", "content_id": "9d771ec9-...", "creator_id": "bob", "status": "classified", "attribution": "likely_ai", "ai_score": 0.718, "confidence": 0.718, "signals": {"llm_judge": 0.82, "stylometric_heuristics": 0.5036, "phrase_pattern_heuristic": 0.7847}}
{"timestamp": "2026-06-30T23:07:46.308527+00:00", "event_type": "submission_classified", "content_id": "672c33ec-...", "creator_id": "carol", "status": "classified", "attribution": "uncertain", "ai_score": 0.4582, "confidence": 0.5418, "signals": {"llm_judge": 0.6, "stylometric_heuristics": 0.5274, "phrase_pattern_heuristic": 0.0}}
{"timestamp": "2026-06-30T23:07:46.900185+00:00", "event_type": "appeal_submitted", "content_id": "672c33ec-...", "creator_id": "carol", "status": "under_review", "original_attribution": "uncertain", "original_ai_score": 0.4582, "original_confidence": 0.5418, "appeal_reasoning": "I am an economics PhD student and wrote this from my own research. ## Known limitations
```

**Formal human writing is the system's worst case.** Academic, legal, or financial prose — like `carol`'s monetary-policy paragraph — uses long, uniform sentences and formal transitions. That pattern lights up *two of the three signals*: the stylometric signal reads the uniformity as AI-like (`0.53`), and the LLM judge reads the formal register as "generic" (`0.60`). Only the phrase-pattern signal (`0.0`, because the writer avoided stock AI clichés) pulls the score back down to `0.46 → uncertain`. A slightly more formal human could easily tip into a false AI lean. This is a direct consequence of the signals measuring *surface regularity*, which legitimate formal writing genuinely has — there is no statistical property cleanly separating "uniform because AI" from "uniform because trained to write formally."

This is exactly why the thresholds are asymmetric and why the appeal path exists: on a writing platform, **falsely accusing a human is worse than missing an AI**, so the system gives borderline formal writing the benign `uncertain` label rather than `likely_ai`, and gives the creator a way to contest it.

A second weak spot: **lightly edited AI text sprinkled with casual markers** ("honestly," "ok so") gets pulled toward `likely_human` by the phrase-pattern signal — a deliberate false-negative we accept under the same asymmetry.

## Spec reflection

**One way the spec helped:** Writing the three label variants *verbatim in [planning.md](planning.md) before any code* meant the label generator was a trivial lookup by the time I built it — the hard design work (plain language, no jargon, never claiming proof) was already done and just needed wiring up. The spec turned a UX problem into a copy-paste.

**One way the implementation diverged:** [planning.md](planning.md) originally specified a `0.75` threshold for `likely_ai` and `0.25` for `likely_human`. Real testing proved `0.75` was unreachable — the stylometric signal caps near `0.55` on short text, so even extreme AI slop only hit `0.72`. I lowered `likely_ai` to `0.70` and widened the human band to `0.40`, which also made the false-positive asymmetry stronger and more intentional. The spec's symmetry was a guess made before I had real score distributions; the implementation corrected it, and planning.md was updated to match.

## AI usage

**1. Generated the Flask skeleton + LLM signal (Milestone 3).** I gave an AI tool my [planning.md](planning.md) detection-signals section and the architecture diagram and asked for the `/submit` route plus the Groq judge function. I **overrode** its prompt design — its first version asked the model for a yes/no verdict; I changed it to return a calibrated `ai_score` 0–1 with a reason, because a binary from the LLM would have defeated the whole "communicate uncertainty" goal. I also added the graceful-degradation path (neutral `0.50`) when no API key is present, which it had not included.

**2. Generated the ensemble scoring function (Milestone 4).** I asked for a weighted combiner from the uncertainty section. The generated thresholds *looked* reasonable but used the planning.md values (`0.75`/`0.25`) that my own testing later proved unreachable — a good example of the spec's warning that "AI tools sometimes implement reasonable-looking scoring that silently diverges." I **caught this by printing each signal score separately**, then recalibrated the thresholds to `0.70`/`0.40` myself and re-tested all five inputs to confirm every label was reachable.

> A bug worth noting: the originally generated LLM-prompt f-string nested triple-quotes (`"""{text}"""` inside an `f"""..."""`), which is a hard `SyntaxError` that prevents the app from even starting. I fixed it by wrapping the text in plain `<<<TEXT ... TEXT` markers instead — a reminder that AI-generated code must be *run*, not just read.

## Stretch feature: ensemble detection

The required minimum is two signals; this project ships **three** with a documented weighting scheme (`0.50` LLM / `0.30` stylometric / `0.20` phrase-pattern — see [Confidence scoring](#confidence-scoring)). The third signal (phrase-pattern) is a genuinely distinct *lexical* dimension on top of the semantic (LLM) and structural (stylometric) signals, and the audit log records all three individual scores on every decision so the ensemble's reasoning is fully transparent and auditable.
